# グリーンフィールド理想設計（ERP4 再設計案）

- status: draft
- updatedAt: 2026-03-03
- scope: ERP4 全体（業務ドキュメント/承認ワークフロー/チャット/通知/監査）

## 目的
仕様書を整理した前提で、現行実装の制約に引きずられず「最終的にこうしたい」理想設計を言語化する。
このドキュメントは、以下の用途を想定する。

- 設計レビューの叩き台（合意形成・優先順位付け）
- 既存実装のリファクタリング/置換のターゲット（到達点の定義）
- 技術選定・運用設計・テスト設計の方針統一

## 前提（仮定）
現時点では組織規模/運用体制/規制要件が確定していない前提で、以下を仮定する。

- 複数プロジェクトを並行運用し、プロジェクト階層（親子/子孫）が存在する
- 証跡（エビデンス）・監査ログ・権限制御が重要
- PoC/試験稼働から開始し、段階的に対象ドメインを拡大する
- 単一VPS（例: Ubuntu + Podman）でも稼働でき、将来的に拡張可能な構成を目指す

上記が成立しない場合（例: 厳格なデータ分離や高い可用性SLOが必須など）、設計判断は変わる。

## 目標（ゴール）
- **横断機能の統一**: ワークフロー（承認/ack/ロック）、通知、監査、参照リンク（内部参照/外部URL）を共通基盤として提供
- **境界の明確化**: ドメインごとの実装差分を「例外」に押し込み、標準パターンを増やす
- **運用耐性**: 管理者操作・例外処理を「想定内」として設計し、理由/監査/再現性を担保
- **監査可能性**: 重要操作は監査ログ・証跡パックに集約し、後追いが容易
- **拡張性**: 新しい書類種別や承認パターンを設定追加で増やせる（コード変更最小化）

## 非目標（このドキュメントでは扱わない）
- 既存データの移行手順の詳細（別途 runbook/移行設計で扱う）
- 給与計算や法令対応の網羅（HR領域の詳細は別ドキュメントへ分離）
- 過度なマイクロサービス化（運用コストが見合わない場合は採用しない）

## 設計原則
- **Modular Monolith First**: 境界は強く、デプロイ単位は小さく始める（必要に応じて分割可能）
- **Configurable Workflow**: 承認・ack・ロックの“規則”は設定（DB/DSL）に寄せる
- **Append-only Audit**: 監査ログは原則追記のみ（論理削除/修正も履歴として残す）
- **Evidence as First-class**: 証跡は添付ではなく「参照」と「スナップショット」を体系化する
- **Policy as Code-ish**: 権限はルールを明文化し、API/バッチ/管理操作で一貫して強制する
- **Fail-safe**: 権限/参照/外部送信は安全側に倒す（許可しない・送らない・遮断する）

## 1. 論理アーキテクチャ（理想）

### 1.1 バウンデッドコンテキスト（モジュール分割）
単一アプリでも、コードとデータの責務を分割し、依存方向を固定する。

| モジュール | 主要責務 | 主なデータ | 代表API/機能 |
| --- | --- | --- | --- |
| Identity & Access | 認証/認可、ユーザ/グループ、SCIM、ポリシー評価 | users, groups, memberships, sessions | OIDC/JWT, SCIM, policy decision |
| Org & Project | 組織、プロジェクト階層、メンバー | orgs, projects, project_tree, project_members | プロジェクト管理、階層検索 |
| Master Data | 顧客/業者/税率/勘定科目等 | customers, vendors, tax_rates, accounts | マスタCRUD、参照制約 |
| Documents | 見積/請求/発注/仕入/経費/休暇/工数などの業務文書 | documents + per-kind details | 作成/更新、PDF、添付 |
| Workflow | 申請/承認/差戻し/却下、ack required、ロック | workflow_definitions, workflow_instances, tasks | 設定、進捗、ガード |
| Evidence & References | 内部参照/外部URL、証跡パック、監査連携 | reference_links, evidence_packs | 参照追加、スナップショット |
| Chat | ルーム、投稿、メンション、添付、ack required UI | rooms, messages, reactions, acks | ルーム権限、検索、証跡参照 |
| Notifications | App通知、メール、Push、ダイジェスト、ミュート | notifications, deliveries, preferences | 通知生成/配信、設定 |
| Integrations | 外部PDF、S3、Slack/Webhook、Google Drive等 | integration_configs, outbox | 送信/同期、SSRF対策 |
| Ops | バッチ、バックアップ、監視、移行 | job_runs, audit_ops | runbook、自動検証 |

分割の狙いは「責務の境界を固定し、実装差分の拡散を止める」ことにある。

### 1.2 デプロイ（理想の物理アーキテクチャ）
当初は「1プロセス+ワーカー」でも運用でき、段階で分割できる形を理想とする。

```
[Browser/Mobile]
   |
   | HTTPS
   v
[Edge/Reverse Proxy]  (rate limit / auth redirect / headers)
   |
   v
[Web App (BFF+API)]  ---[Outbox/Queue]--> [Worker(s)]
   |
   +--> [PostgreSQL] (primary)
   +--> [Object Storage] (S3互換: PDF/添付/証跡パック)
   +--> [Redis] (cache / rate limit / job locks)  ※必須ではない
   +--> [Observability] (logs/metrics/traces)
```

将来の分割は、モジュール単位で「API境界・イベント」を設計しておくことで可能にする。

## 2. コア設計（横断機能）

### 2.1 IDと参照の設計
- すべての主要エンティティは **機械的ID**（例: ULID/UUID）を主キーとする
- UI表示名は別カラム（変更可能）とし、参照はIDを基準に安定させる
- 外部URLは「文字列」ではなく **正規化されたURLエンティティ**として保持する

### 2.2 監査ログ（Audit）
監査ログは「誰が/いつ/何を/なぜ（理由）/どの権限で」実行したかを追えることを必須とする。

- append-only（原則追記のみ）
- actor 種別: user / admin / system / agent
- admin 操作は必ず区別し、理由（free text）を必須化できる
- 重要イベントは evidence pack と紐づけ可能

### 2.3 イベント駆動（Outbox）
通知・外部送信・監査・集計を「同期APIの副作用」に依存させない。

- DBトランザクション内で outbox にイベントを追記
- ワーカーが outbox を処理（リトライ/冪等）
- 外部送信は allowlist/サイズ制限/タイムアウト等を標準化

### 2.4 権限制御（Policy）
理想は「ロール固定」より **グループ中心**のアクセス制御に統一する。

- グループは SCIM 由来に加えてアプリ内作成も可能（運用に合わせる）
- ルーム/文書/プロジェクト等のリソースに対し、閲覧/投稿/編集/承認/管理 をポリシーで判定
- ポリシー適用は API 層で強制し、UIは結果を反映するだけに寄せる

## 3. ワークフロー（承認/ack/ロック）の理想設計

### 3.1 目的
見積/請求/発注/仕入/経費/休暇/工数等のドメインが増えても、ワークフロー基盤は共通のまま運用できる状態を目標とする。

### 3.2 モデル（概要）
- `WorkflowDefinition`（設定）: バージョン管理された定義（変更は新バージョンとして追加）
- `WorkflowInstance`（実行）: 1つの対象（subject）に対する進行中のインスタンス
- `Task`（タスク）: 承認/確認/ack required 等の実行単位
- `Guard`（ガード）: 状態遷移の前提条件（証跡必須、理由必須、権限必須など）
- `Lock`（ロック）: 編集可能範囲の制御（状態/権限/例外を統一的に表現）

### 3.3 定義DSL（例）
定義は DB に JSON/YAML として保持し、UIで編集可能にする（監査・レビューは必須）。

- ステップ種別:
  - sequential（順序固定）
  - parallel_all（全員承認）
  - parallel_any（誰か1人承認）
  - group_any / group_all（グループ承認、順不同）
- 期限:
  - dueAt は「リマインド基準」と「可視化」を主用途にし、自動クローズは原則しない（誤動作が高コスト）
- 例外:
  - admin/mgmt override は定義で許容可否を制御し、理由必須 + 監査ログ必須

### 3.4 対象（subject）と統合ポイント
ワークフローは「文書」だけでなく「高リスク操作」や「設定変更」にも適用できる。

- subject 型の例:
  - document:expense / document:vendor_invoice / document:leave_request ...
  - action:high_risk_api (例: 支払確定、リンク解除、管理者操作)

### 3.5 UI（理想）
- 申請画面の中で、必要な証跡（チャット/添付/URL）を選択できる
- 承認画面で、証跡が「一覧・プレビュー・リンク」で確認できる
- 例外操作（管理者更新）は通常操作と視覚的に区別される

## 4. 証跡（エビデンス）と参照リンクの理想設計

### 4.1 参照リンク（ReferenceLink）
メモ本文に自由記入がある場合でも、機械可読な参照を別枠で保持する（監査・検索・整合性のため）。

- `ReferenceLink`
  - kind: `external_url` / `internal_ref`
  - label: 人間向けラベル（任意）
  - target:
    - external: URL（バリデーション後にリンク化、複数可）
    - internal: `{entityType, entityId}`（候補検索+部分一致）
  - visibility: 参照の可視範囲（権限と連動）

### 4.2 証跡パック（Evidence Pack）
監査や外部提出に耐える形で、対象の意思決定に必要な情報をスナップショット化する。

- 構成:
  - 参照リンクの一覧
  - チャット投稿の参照（messageId）+ 表示用スナップショット（改ざん耐性）
  - 添付ファイル（必要ならハッシュ）
  - ワークフロー履歴（誰が/いつ/結果/理由）
- 保存先:
  - ローカル/オブジェクトストレージ（S3互換）を抽象化

## 5. チャットの理想設計（エビデンス用途を含む）

### 5.1 ルームとスコープ
- ルームIDは機械的に割り振り、表示名は変更可能
- 権限は「内部/外部」固定ロールではなく、グループに対する閲覧/投稿権限で統一
- 個人総務チャットは社員登録時に常設生成（事務連絡のセーフティネット）

### 5.2 エビデンス参照
- 任意のルーム/任意の投稿は URL/ID により参照可能（権限がある範囲）
- 申請画面内で「投稿を検索して選択」できる（画面遷移を最小化）
- 参照は「投稿へのリンク + 手動引用（テキスト）」を許容（部分参照の現実的運用）

---

## 6. 通知の理想設計
- App通知（一覧）を基盤とし、メール/Push は配送チャネルとして扱う
- ユーザは「ルームごと」に通知を設定可能:
  - 全投稿 / 自分メンションのみ / ミュート
  - ミュートは期間指定（例: 10分/1時間/今日だけ）
- メール通知はリアルタイム以外に「一定間隔でまとめて」を選択可能（既定例: 10分）

## 7. 技術スタック（フリーハンド再考）

### 7.1 結論（推奨）
不確定要素が多い場合の推奨は「**Modular Monolith + Postgres + 型付きAPI契約（OpenAPI等）**」を基軸にする。
言語/フレームワークはチームスキルと運用要件で最適解が変わるため、以下の判断軸で選定する。

| 判断軸 | 重要度の例 | 主に効く選択 |
| --- | --- | --- |
| PoC速度 | 試験稼働を急ぐ | TypeScript/Node, 生成コード活用 |
| 長期保守 | 数年単位で運用 | Kotlin/Go, SQL中心設計、移行容易性 |
| 統制/監査 | 監査対応が重い | 監査ログ/証跡/ロック/権限の基盤化（言語より設計が重要） |
| 採用容易性 | 人員入替が多い | 一般的スタック（TS/React等） |

### 7.2 候補スタック（例）
実際の採用はチームスキルと運用要件で決める必要がある。

#### A) TypeScript フルスタック（推奨）
- Backend: Node.js LTS + Fastify（または NestJS）
- ORM: Prisma（または SQL/Query Builder へ段階移行）
- Frontend: React +（Next.js か Vite）
- Schema: OpenAPI を真にし、型生成でフロントと同期
- Job: 同一コードベース内の worker（outbox 消費）

#### B) Kotlin（堅牢性重視）
- Backend: Kotlin + Spring Boot
- DB: Flyway + jOOQ（SQL中心）
- Pros: 型安全・並行性・長期運用の保守性
- Cons: PoC速度は落ちやすい、周辺ツール整備が必要

#### C) Go（運用単純化重視）
- Backend: Go（標準HTTP + 生成コード）
- Pros: 単一バイナリ、パフォーマンス、依存が軽い
- Cons: ドメインロジックの抽象化/DSL/UI連携に設計力が必要

### 7.3 参考: 既存資産を最大限活かす場合（現実解）
ゼロからの技術選定とは別に、既存実装・CI・E2E・運用スクリプトを活用して段階導入する場合は、
TypeScript/Node を維持しつつ「境界強化」「横断基盤の統一」に集中するのが費用対効果が高い。

### 7.4 インフラ（VPS起点）
- 試験稼働: Ubuntu + Podman + systemd + reverse proxy（Caddy/Nginx）
- 監視: Prometheus/Grafana または Sentry 等のSaaS
- 重要: secrets はリポジトリ外（env file/secret manager）で管理

## 8. 追加で決めるべき論点（設計の分岐点）
- マルチテナントの分離レベル（DB分離/スキーマ分離/RLS）
- 監査ログ・証跡パックの保管期間/削除ポリシー（法務・統制要件）
- 参照リンクの可視化範囲（休暇理由などのプライバシー設計）
- ワークフロー定義の変更適用ルール（新規のみ/進行中に適用/例外）
- 検索基盤（Postgres FTS で十分か、専用検索が必要か）

## 9. 次のステップ（提案）
- 仕様書（現行整理版）に対して、この理想設計の「差分/不足/過剰」を付箋化する
- ワークフロー基盤（定義/instance/lock/guard/evidence）を最優先の統一対象としてロードマップ化する
- 既存実装の改善は「理想設計のどこに寄せるか」を明記してから着手する
