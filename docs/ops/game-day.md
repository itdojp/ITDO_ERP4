# ゲームデイ（演習）計画（暫定）

## 目的
机上/演習を通じて、障害対応手順の実効性を検証し、改善点を継続的に Issue 化する。

## 実施頻度（暫定）
- まずは **四半期に1回**（PoC/初期運用の間は必要に応じて追加）

## 参加者（例）
- 運用担当（実施責任者）
- 管理者/マネージャ（意思決定）
- 開発担当（原因切り分け/修正）

## 事前準備
- `docs/ops/incident-response.md` を参加者に共有
- アラート/ダッシュボード定義（`docs/ops/alerting.md` / `docs/ops/dashboard.md`）を確認
- 演習用の「検証環境」を用意（Podman想定）

## 想定シナリオ（例）
### シナリオA: DB接続不可
- 期待検知: `/readyz` が 503
- 目標: 依存障害として切り分け、復旧までの手順が破綻しない

### シナリオB: デプロイ不整合（frontend/backend の組み合わせ不整合）
- 期待検知: 主要導線で 5xx/不整合が増える
- 目標: 直近デプロイSHA確認→ロールバック判断まで到達

### シナリオC: 外部依存（メール送信/AV等）の停止
- 期待検知: ジョブ失敗が増える、または機能の一部が停止
- 目標: 影響範囲を限定し、代替策（キュー/リトライ/機能停止）を判断できる

## 成功条件（最小）
- 検知→一次切り分け→復旧（またはロールバック）まで到達できる
- 対応中の記録（時系列）が残る
- 改善点が **最低1件以上** Issue 化される

## 実施後
- `docs/ops/postmortem-template.md` を使って振り返りを作成（演習でも作成する）
- 再発防止/改善を Issue 化し、担当者/期限を設定する
